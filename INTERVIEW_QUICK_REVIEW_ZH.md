# CityScout 面试速记（支付 / 防超卖 / 高并发）

## 1. 30秒自我介绍版（可直接背）

我在这个项目里做的是秒杀券下单链路优化，核心是 Redis + Lua 做原子资格校验，再异步落库。
Lua 一次性完成库存判断、一人一单判断、预扣库存，减少数据库在高并发下的压力。
数据库侧再用 `stock > 0` 的 CAS 更新做最终兜底，防止超卖。
目前我也发现了生产级风险：异步事务代理调用方式、入队失败补偿、支付回调闭环等，我整理了分阶段改造方案。

## 2. 2分钟技术说明版

当前链路：

1. 用户请求 `POST /voucher-order/seckill/{id}`。
2. 服务执行 `seckill.lua`：
   - 判断 Redis 库存是否足够。
   - 判断用户是否已下单。
   - 原子扣减 Redis 库存并写入下单用户集合。
3. 成功后生成订单ID，放入阻塞队列。
4. 单线程异步消费者落库：
   - MySQL 扣减库存 `stock = stock - 1 and stock > 0`。
   - 保存订单。

为什么这么设计：

- Redis 原子脚本把热点竞争放在内存层，吞吐高。
- 异步化削峰，保护数据库。
- DB CAS 是最终一致性兜底，避免超卖。

## 3. 你这个项目的“亮点 + 问题”答法

亮点：

- 有 Redis Lua 原子门闸，避免“查库存+扣库存”并发穿透。
- 有数据库 CAS 兜底，形成双保险。
- 有异步化思路，前台响应快。

问题（面试加分点，主动说）：

- 异步线程里用 `AopContext.currentProxy()` 可能拿不到代理，事务可能失效。
- Lua 成功后如果队列满，`add()` 失败没有补偿，会造成库存与订单不一致。
- 目前没有完整支付回调和超时关单释放库存闭环。
- `tb_voucher_order` 缺少 `(user_id, voucher_id)` 唯一索引，DB 终态约束不够硬。
- 本地内存队列不持久，服务重启可能丢单。

## 4. 高频面试题（短答案）

Q1：你怎么防超卖？
A1：三层防线。第一层 Redis Lua 原子扣库存；第二层一人一单集合去重；第三层 MySQL `stock > 0` CAS 最终兜底。

Q2：为什么不用纯数据库锁？
A2：秒杀峰值下数据库会成为瓶颈，锁竞争严重。Redis 在内存执行 Lua 原子脚本，吞吐更高，适合做前置拦截。

Q3：为什么 Redis 扣了还要 DB 再扣一次？
A3：Redis 是流量闸门，DB 是最终账本。DB 再扣一次是最终一致性校验，防止 Redis 与 DB 短暂不一致导致超卖。

Q4：一人一单怎么保证？
A4：Redis Set 快速判重 + 数据库唯一索引 `(user_id, voucher_id)` 作为最终强约束。

Q5：如果支付没成功怎么办？
A5：订单状态机 + 支付回调幂等 + 超时关单任务释放库存，形成“预占-支付-确认/释放”的闭环。

Q6：高并发下怎么避免消息丢失？
A6：用持久化消息系统（Redis Stream/RabbitMQ/Kafka）替代本地队列，配合 ACK、重试、死信队列。

## 5. 可落地改造路线（面试可说）

第一阶段（1-2天）：

1. 修正异步事务调用方式（改成注入代理Bean或拆分事务服务）。
2. `add()` 改 `offer()`，入队失败执行补偿 Lua 回滚预扣。
3. Lua 增加库存key不存在保护。

第二阶段（3-5天）：

1. 加唯一索引 `(user_id, voucher_id)`。
2. 补秒杀时间窗口校验（Lua 或 Redis 活动元数据）。
3. 增加超时未支付关单与库存释放。

第三阶段（1-2周）：

1. 本地队列升级为持久化消息流。
2. 接支付回调并做幂等状态迁移。
3. 建立对账与补偿任务。

## 6. 一句话总结（收尾）

这套设计思路是对的：用 Redis 扛并发、用 DB 保终态；但要达到生产级，关键是把“幂等、补偿、持久化、支付闭环”补完整。

## 7. 你可以背的关键词

- 原子性：Lua 一次执行
- 最终一致性：Redis 闸门 + DB 账本
- 幂等：回调去重、状态机防重复推进
- 补偿：入队失败回滚预扣
- 持久化：MQ/Stream + ACK
- 防超卖：Lua + CAS + 唯一索引
